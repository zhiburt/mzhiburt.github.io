About 'std::mem::transmut'ation

"I'm particularly interested in Transfiguration, you know, turning something into something else, of course, it's supposed to be very difficult —" — Hermione Granger regarding transformation [src]

The do

There's a type csv::StringRecords which doesn't implement IntoIterator but &csv::StringRecords does.

I was thinking if we would keep the structure and the reference along themeslfs, it would be fine. But I know that rust does not allow that (yet, cause it's self-referenctial struct). So I was thinking maybe to cast &'a into &'static and be fine.

Structure

struct CsvStringRecord {
    record: StringRecord,
    iter: Option<StringRecordIter<'static>>,
}

Here's a building function

impl CsvStringRecord<'static> {
    fn new(record: StringRecord) -> Self {
        let mut t = Self { record, iter: None };

        let iter = t.record.iter();
        let static_iter =
            unsafe { std::mem::transmute::<StringRecordIter<'_>, StringRecordIter<'static>>(iter) };

        t.iter = Some(static_iter);

        t
    }
}

But it doesn't work.....
It comes with a stable SIGSEGV panic.

Why it doesn not work?

Apparently the problem was in a .iter() call. We take a refence local to a CsvStringRecord::new function.

impl<'a> IntoIterator for &'a StringRecord {
    type IntoIter = StringRecordIter<'a>;
    type Item = &'a str;

    #[inline]
    fn into_iter(self) -> StringRecordIter<'a> {
        StringRecordIter(self.0.iter())
    }
}

impl<'r> IntoIterator for &'r ByteRecord {
    type IntoIter = ByteRecordIter<'r>;
    type Item = &'r [u8];

    #[inline]
    fn into_iter(self) -> ByteRecordIter<'r> {
        ByteRecordIter {
            // <- RIGHT HERE we get the reference to a stack,
            // but the reference is no more valid when we leave a `new` function,
            // so we are referencing an invalid address
            r: self,
            last_start: self.as_slice().len(),
            last_end: 0,
            i_forward: 0,
            i_reverse: self.len(),
        }
    }
}

Conclusion

Be carefull when doing unsafe.

It would work if the &'a reference was against a buffer itself via pointer but not a reference to a sub structure.

For example that's how we could get such a reference for &str.

unsafe {
    str::from_utf8_unchecked(slice::from_raw_parts(text.as_ptr(), text.len()))
};

But we can't do it for csv::StringRecords cause the buffer is private.

Take care.
